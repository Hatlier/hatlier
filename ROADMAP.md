## いまの全体像（見直し 2026-07-22）

MVPの「直す → 保存 → 書き出す」はだいたい揃った。  
これから並列で進む大きな軸は次の3つ（**同列で検討**、実装順は手触りのあとで決めてよい）。

| 軸 | ひとこと |
|---|---|
| **A. 手触り（MVP-D）** | 迷わない・壊れない・言ってることが分かる |
| **B. フォーマット拡張** | 縦スクロール一択を超え、レスポンシブを保ったまま表現の自由度を上げる |
| **C. 出力プロファイル** | 作り始めに「どんなHTMLにしたいか」を選び、部品・テンプレをそれに合わせて制限する |

**マイテンプレート**は B/C と同じレイヤ（再利用・フォーマットの延長）。置き場導線（Drive等）より、フォーマット議論とセットで扱う。

## Track B — フォーマット拡張（デザイン自由度）

### 問題意識

いまの契約は **ブロックの縦積み** が強い。レスポンシブは保ちやすいが、

- 横スクロール帯
- カルーセル
- サイドバー
- ハンバーガー／モバイルナビ

といった「ページとしての構図」が出しにくい。

### 方針（案）

1. **レスポンシブは捨てない**（モバイルで壊れないことを契約に残す）
2. 自由配置キャンバスにはしない（Figma化しない — 制約で守る思想は維持）
3. 「ブロック」に加え、**レイアウト枠（region / shell）** をフォーマットに足す方向を検討する  
   例: `shell`（ヘッダー＋メイン＋サイド）、`carousel`、`hstack`（横並びセクション）
4. 既存の縦積みドキュメントは **互換のまま開ける**（バージョン付きフォーマット）

### 検討バックログ（実装は仕様合意後）

- [ ] フォーマット v2 の骨子（`data-hatlier-version` / shell / region）
- [ ] 部品: カルーセル、タブ、サイドバー付きシェル、ハンバーガー
- [ ] 横スクロール・セクション（レスポンシブ時は縦に折るルール）
- [ ] AIプロンプトを v2 構造に追従
- [ ] マイテンプレ（下と一体）

## Track C — 出力プロファイル（作り始めの制約）

### 問題意識

書き出し形式や公開先で **許されるものが違う**。  
後から「CSP強いホストに上げたらスクリプトが動かない」より、**最初にプロファイルを選ばせ、部品パレットとテンプレをそれでフィルタする**方が UX として正しい。

### プロファイル案（名前は仮）

| ID | 想定 | 制限のイメージ |
|---|---|---|
| `safe-static` | ほぼどこでも・メール添付寄り | JSほぼなし、外部fetchなし |
| `csp-strict` | CSPの厳しい静的ホスト | インラインJS制限、評価系・録音・一部ウィジェット不可 |
| `interactive` | 通常の1枚HTML | トグル/FAQ/グラフなど現状どおり |
| `rich` | 拡張部品あり（fetch可） | 後段の占い・ゲーム等 |

### UX 案

1. **新規**の最初（またはテンプレ選択の前）にプロファイルを選ぶ
2. 選んだプロファイルを `doc.profile`（書き出しHTMLの `data-profile`）に保存
3. 部品パレット・テンプレ・AIプロンプトが **使えないものを出さない／追加できない**
4. プロファイル変更は可能だが、「使えなくなる部品がある」と明示して確認

### 部品へのタグ付け（実装の前提）

各 `REG` エントリに例えば:

- `needs: [] | ["js"] | ["js","media"] | ["fetch"]`
- プロファイルが許可しない `needs` を持つ部品はパレット非表示＋既存文書では警告
