## 0. このメモの目的

いまの Hatlier は **ブロックの縦積み（フォーマット v1）** で「崩れない・直せる」を実現している。  
一方で次が足りない、という認識を仕様にする:

1. **デザイン表現の自由度**（横の構図・ナビ・カルーセル等）を、レスポンシブを捨てずに上げたい  
2. **作り始めに出力の制約（プロファイル）** を選び、部品・テンプレ・AI出力をそれに合わせたい  
3. **マイテンプレ** を、プロファイル付きの再利用として同じ土台に乗せたい  

自由配置キャンバス（Figma化）にはしない。制約で守る思想は維持する。

---

## 1. 現状（フォーマット v1）の要約

| 項目 | v1 |
|---|---|
| 文書の形 | `theme` / `bg` / `font` + `blocks[]` |
| レイアウト | ブロックを上から下へ |
| 編集データ | `#hatlier-doc`（JSON）または `.blk` DOM |
| 強み | 単純・再編集しやすい・AIプロンプトが書きやすい |
| 弱み | ページ構図（サイドバー・横スクロール・カルーセル・ハンバーガー）が出しにくい |

v2 は v1 を捨てず、**拡張する**。

## 2. フォーマット v2 — 骨子

### 2.1 バージョン

書き出し HTML / JSON に明示する:

```html
<body class="doc" data-hatlier-version="2" data-profile="safe" …>
```

```json
{
  "version": 2,
  "profile": "safe",
  "theme": "paper",
  "bg": "plain",
  "font": "elegant",
  "shell": { "type": "stack", "regions": { … } },
  "blocks": [ … ]
}
```

- **読取:** v1（version 無し / 1）は従来どおりブロック縦積みとして開く  
- **書出:** エディタが v2 対応後は `version: 2` を基本にする（中身が v1 相当でも可）

### 2.2 二つの層

```
Shell（構図）     … ページの骨格。レスポンシブの折り方を持つ
  └ Regions      … 枠（main / side / header / …）
       └ Blocks  … 従来どおりの部品（hero, text, …）
```

| 層 | 役割 | 例 |
|---|---|---|
| **Shell** | どこに何を置くか | `stack`（現行相当）, `sidebar`, `topnav` |
| **Region** | 枠の中身の列 | `main`, `side`, `drawer` |
| **Block** | 編集単位の部品 | 既存 REG + 新規（carousel 等） |

v1 文書は「shell = stack, region = main のみ, blocks = 従来の配列」とみなせる。

### 2.3 Shell タイプ（最小セット案）

| shell.type | デスクトップ | モバイル（必須ルール） |
|---|---|---|
| `stack` | 縦一列（v1互換） | 同じ |
| `sidebar` | 左（または右）サイド＋メイン | サイドはハンバーガー／ドロワーへ |
| `topnav` | 上部ナビ＋メイン | ナビはハンバーガーへ折畳み |

**やらない（v2初期）:** ピクセル自由配置、重ね順序のドラッグ、絶対座標。

### 2.4 新規ブロック／部品（表現の自由度）

部品は「ブロック」として Region に載せる。候補の優先順:

| 優先 | type（仮） | 内容 | 実装方針 |
|---|---|---|---|
| P0 | `carousel` | 横スライドのカルーセル | **CSS scroll-snap 優先**（CSP・無難プロファイルと両立） |
| P0 | `tabs` | タブ切り替え | 可能なら CSS / `details`。難しければ `standard` 以上で軽量JS |
| P1 | `nav` | ページ内リンク一覧／アンカー | 原則JS不要 |
| P1 | （shell連携） | ハンバーガー／ドロワー | shell 側。`safe` では開閉を CSS で、難しければ `standard` |
| P2 | `hstack` | 横並びセクション（狭幅では縦） | CSSのみ |

部品は今後かなり増える想定。追加UIは現状どおり「挿入したい場所 → 部品を選ぶ」。  
**プロファイルによる出し分けも、この部品選択の段階で行う**（使えない部品は出さない／理由を短く示す）。

### 2.5 レスポンシブ契約（捨てない）

1. すべての shell は **モバイルで情報が欠けない**（ドロワー内に退避は可）  
2. 横スクロール帯は「意図した領域だけ」。ページ全体の横スクロール強制は禁止  
3. タッチで主要操作が完結する  
4. 印刷／PDF では shell を `stack` 相当に展開してよい（プロファイルで制御も可）

### 2.6 編集UX（イメージ）

- キャンバス上で「構図（shell）」を選ぶ／切り替える（データは保持できる範囲でマージ）  
- 部品パレットは **今フォーカスしている region** に挿入  
- サイドバーとメインでブロックをドラッグ移動できると望ましい（初期はコピー＋削除でも可）

## 3. 出力プロファイル（合意反映）

### 3.1 ねらい

「あとから CSP に弾かれる」を防ぐ。  
思想の中心は **いちばん無難（超強い CSP 環境でも動く出力）をデフォルトにする** こと。

ユーザが選ぶときは技術用語で迷わせない。まずは **強／弱（＋必要なら「いままでどおり」）** 程度の粒度。

### 3.2 プロファイル定義（合意後）

CSPまわりは **実質2モード**。表示名は仮。

| profile ID | ユーザ向けの呼び方（仮） | デフォ | 許容 |
|---|---|---|---|
| `safe` | **いちばん無難**（CSPが厳しくても動く） | **✓ デフォルト** | HTML + CSS のみ。**JS なし**。外部 fetch なし。今日の「超強い CSP」でも通る思想 |
| `standard` | **ややゆるめ**（ふつうの静的ホスト向け） | | 軽量JS可（トグル・FAQ・グラフなど現行相当）。fetch は原則なし。「ちょっとした CSP」には乗る想定 |
| `rich` | 拡張（あとで） | Later | fetch 部品・重いウィジェット。初期UIには出さなくてもよい |

旧メモの `safe-static` / `csp-strict` / `interactive` は、上の **`safe` / `standard`** に寄せて整理する（`interactive` ≒ `standard`）。

文書への保存例:

- JSON: `"profile": "safe"`
- HTML: `data-profile="safe"`

### 3.3 部品のタグ（実装の前提）

各部品定義（いまの `REG`）にメタを足す:

```js
{
  name: "トグル",
  needs: ["js"],          // 空配列なら HTML/CSS のみ
  shells: ["*"],
}
```

| needs | 意味 |
|---|---|
| （なし） | 静的で完結 → `safe` でも可 |
| `js` | 書き出しにスクリプトが必要 → `standard` 以上 |
| `media` | カメラ／録音など → `standard` 以上 |
| `fetch` | 実行時ネットワーク → `rich` |

**フィルタ規則**

| profile | 許可する needs |
|---|---|
| `safe` | なしのみ（**一切 JS 禁止**） |
| `standard` | なし, `js`, `media` |
| `rich` | すべて |

パレット（挿入場所クリック → 部品選択）に **出せるものだけ出す**。テンプレ・AIプロンプト例も同様。  
プロファイルを厳しく変更するときは、非互換部品のリストを出して確認。

### 3.4 新規フロー（UX）— 合意

```
新規
  → テンプレ or 白紙（マイテンプレ含む）
  → 編集（デフォルト profile = safe）
  → 書き出し
```

- **プロファイル選択は必須にしない（スキップ可）**  
- デフォルトは **`safe`（いちばん無難）**  
- どこかでサジェスト（「ホスト先が厳しいなら無難のまま」等）は eventual。急がない  
- 変更UIは設定またはデザイン近傍。「厳しくすると消える部品がある」を明示

### 3.5 AIプロンプトとの関係

- `safe` → script 禁止・インラインハンドラ禁止を絶対ルールに  
- `standard` → 現行に近い許可範囲  
- 列挙するブロックはフィルタ後のみ

### 3.6 書き出し形式との関係（将来）

プロファイルは「文書の能力」、HTML/ZIP/PDF は「袋」。初期は案内のみでロックしない。

## 4. マイテンプレート（同列）— 合意

| 項目 | 合意 |
|---|---|
| 保存先（初期） | **localStorage のみでよい** |
| 上限 | **件数固定ではなく、使用メモリ量ベース**を基本にする |
| ユーザ設定 | 上限などを変えられるとよい → **ブラウザ完結のユーザ設定ページ**が必要になりうる（アカウントなし） |
| 保存内容 | `version`, `profile`, `shell`, `blocks`, theme 等 |
| 一覧 | 新規フローの「公式／マイ」 |
| プロファイル | 保持。開くとき不一致なら確認 |

最小版（名前付きで今の文書を残す）は、設定ページより先に出してもよい。

## 5. 移行・互換

| 方向 | 挙動 |
|---|---|
| v1 → 編集 | shell=`stack` として内部昇格（ユーザには黙ってよい） |
| v1 HTML を開く | 現行どおり |
| v2 HTML を古いエディタで開く | 未知属性は無視。可能なら main region の blocks だけ読むフォールバックを検討 |
| AI が v1 構造を吐く | 引き続き受理（profile 未指定なら `safe`、明示があればそれに従う） |

---

## 6. やらないこと（この草案の境界）

- 自由配置・レイヤー・ピクセル指定のデザインツール化  
- プロファイルなしで全部品を常時解放したまま CSP 対応を「書き出しオプションだけ」でごまかすこと  
- v1 文書を開けなくする破壊的アップグレード  
- いきなり rich 用 fetch 部品から着手すること（先に shell + profile 骨格）

## 7. 実装フェーズ提案（合意後の更新）

| Phase | 内容 |
|---|---|
| **0** | 本メモ合意（本節）。現行 `REG` の `needs` 仕分け表 | **済**（現状すべて safe 可。`needs` 省略＝[]） |
| **1** | profile=`safe` デフォルトを文書に載せる＋パレットフィルタの骨格（UIは控えめ・スキップ可） | **済（0.4.0）** — デザインパネルで説明付き選択 |
| **2** | マイテンプレ最小（localStorage・メモリ上限）／ユーザ設定の芽 | **済（0.5.0）** — 新規モーダル＋デザインに上限KB |
| **3** | shell / carousel（CSS snap）など表現の一本 | **carousel 済（0.5.0）**。shell は次 |
| **4** | `standard`/`rich` の出し分け完成、設定ページ、AIプロンプト追従 | プロファイル骨格は 0.4.0。rich・専用設定ページは残 |
