# Hatlier フォーマット v2 ＋ 出力プロファイル — 仕様メモ（草案）

> 実装前の合意用。1枚で全体像を掴む。確定したら `SPEC.md` に要約を取り込み、詳細は本ファイルを正本にする。  
> 作成: 2026-07-22 / ステータス: **草案（議論中）**

---

## 0. このメモの目的

いまの Hatlier は **ブロックの縦積み（フォーマット v1）** で「崩れない・直せる」を実現している。  
一方で次が足りない、という認識を仕様にする:

1. **デザイン表現の自由度**（横の構図・ナビ・カルーセル等）を、レスポンシブを捨てずに上げたい  
2. **作り始めに出力の制約（プロファイル）** を選び、部品・テンプレ・AI出力をそれに合わせたい  
3. **マイテンプレ** を、プロファイル付きの再利用として同じ土台に乗せたい  

自由配置キャンバス（Figma化）にはしない。制約で守る思想は維持する。

---

## 1. 現状（フォーマット v1）の要約

| 項目 | v1 |
|---|---|
| 文書の形 | `theme` / `bg` / `font` + `blocks[]` |
| レイアウト | ブロックを上から下へ |
| 編集データ | `#hatlier-doc`（JSON）または `.blk` DOM |
| 強み | 単純・再編集しやすい・AIプロンプトが書きやすい |
| 弱み | ページ構図（サイドバー・横スクロール・カルーセル・ハンバーガー）が出しにくい |

v2 は v1 を捨てず、**拡張する**。

## 2. フォーマット v2 — 骨子

### 2.1 バージョン

書き出し HTML / JSON に明示する:

```html
<body class="doc" data-hatlier-version="2" data-profile="interactive" …>
```

```json
{
  "version": 2,
  "profile": "interactive",
  "theme": "paper",
  "bg": "plain",
  "font": "elegant",
  "shell": { "type": "stack", "regions": { … } },
  "blocks": [ … ]
}
```

- **読取:** v1（version 無し / 1）は従来どおりブロック縦積みとして開く  
- **書出:** エディタが v2 対応後は `version: 2` を基本にする（中身が v1 相当でも可）

### 2.2 二つの層

```
Shell（構図）     … ページの骨格。レスポンシブの折り方を持つ
  └ Regions      … 枠（main / side / header / …）
       └ Blocks  … 従来どおりの部品（hero, text, …）
```

| 層 | 役割 | 例 |
|---|---|---|
| **Shell** | どこに何を置くか | `stack`（現行相当）, `sidebar`, `topnav` |
| **Region** | 枠の中身の列 | `main`, `side`, `drawer` |
| **Block** | 編集単位の部品 | 既存 REG + 新規（carousel 等） |

v1 文書は「shell = stack, region = main のみ, blocks = 従来の配列」とみなせる。

### 2.3 Shell タイプ（最小セット案）

| shell.type | デスクトップ | モバイル（必須ルール） |
|---|---|---|
| `stack` | 縦一列（v1互換） | 同じ |
| `sidebar` | 左（または右）サイド＋メイン | サイドはハンバーガー／ドロワーへ |
| `topnav` | 上部ナビ＋メイン | ナビはハンバーガーへ折畳み |

**やらない（v2初期）:** ピクセル自由配置、重ね順序のドラッグ、絶対座標。

### 2.4 新規ブロック／部品（表現の自由度）

部品は「ブロック」として Region に載せる。候補の優先順:

| 優先 | type（仮） | 内容 | JS |
|---|---|---|---|
| P0 | `carousel` | 横スライドのカルーセル（タッチスワイプ可） | 軽量JS or CSS scroll-snap |
| P0 | `tabs` | タブ切り替え | 軽量JS or details 応用 |
| P1 | `nav` | ページ内リンク一覧／アンカー | 原則JS不要 |
| P1 | `drawer-toggle` | ハンバーガー開閉のトリガー表示（shell とセット） | JS |
| P2 | `hstack` | 横並びセクション（狭幅では縦積み） | CSSのみ可 |

`sidebar` / ハンバーガーは **部品単体より shell の振る舞い** として定義する方が破綻しにくい。

### 2.5 レスポンシブ契約（捨てない）

1. すべての shell は **モバイルで情報が欠けない**（ドロワー内に退避は可）  
2. 横スクロール帯は「意図した領域だけ」。ページ全体の横スクロール強制は禁止  
3. タッチで主要操作が完結する  
4. 印刷／PDF では shell を `stack` 相当に展開してよい（プロファイルで制御も可）

### 2.6 編集UX（イメージ）

- キャンバス上で「構図（shell）」を選ぶ／切り替える（データは保持できる範囲でマージ）  
- 部品パレットは **今フォーカスしている region** に挿入  
- サイドバーとメインでブロックをドラッグ移動できると望ましい（初期はコピー＋削除でも可）

## 3. 出力プロファイル

### 3.1 ねらい

「あとから CSP に弾かれる」を防ぐ。  
**新規の最初（テンプレ選択の前または同時）にプロファイルを選ばせ、部品・テンプレ・AIプロンプトをフィルタする。**

### 3.2 プロファイル定義（仮ID）

| profile | 想定用途 | 許容 |
|---|---|---|
| `safe-static` | メール添付・超保守的ホスト | CSS＋HTML。JSなし。外部fetchなし。フォームの高度なことは最小 |
| `csp-strict` | CSPの厳しい静的ホスト | インラインJS・eval不可を前提。必要な挙動はなるべくCSS/`details`等。どうしてもJSが要る部品は除外 |
| `interactive` | 現行デフォルト相当 | トグル・FAQ・グラフ・軽量UI JS可。外部fetchは原則なし |
| `rich` | 拡張・エンタメ | fetch部品・より重いウィジェット可（本体肥大化は別議論） |

文書に保存:

- JSON: `"profile": "csp-strict"`
- HTML: `data-profile="csp-strict"`

### 3.3 部品のタグ（実装の前提）

各部品定義（いまの `REG`）にメタを足す:

```js
{
  name: "トグル",
  needs: ["js"],          // 空配列なら HTML/CSS のみ
  shells: ["*"],          // 載せてよい shell。制限するなら列挙
  // …
}
```

| needs の例 | 意味 |
|---|---|
| （なし） | 静的で完結 |
| `js` | 書き出しHTMLにスクリプトが必要 |
| `media` | カメラ／録音など |
| `fetch` | 実行時ネットワーク |

**フィルタ規則**

| profile | 許可する needs |
|---|---|
| `safe-static` | なしのみ |
| `csp-strict` | なし、または「CSP対応と明示された js」（初期はなしのみでも可） |
| `interactive` | なし, `js`, `media` |
| `rich` | すべて |

パレットに出ない／テンプレに使えない／AIプロンプト例にも載せない。  
既存文書で profile を厳しく変更する場合は、**非互換部品のリストを出して確認**。

### 3.4 新規フロー（UX）

```
新規
  → (1) 出力プロファイル選択
  → (2) テンプレ or 白紙（マイテンプレ含む）
  → 編集（パレットは profile で制限済み）
  → 書き出し（profile を data-profile / JSON に埋め込み）
```

プロファイル変更 UI は設定またはデザイン近傍に置く。「厳しくすると消える部品がある」を必ず言う。

### 3.5 AIプロンプトとの関係

- プロファイルに応じてプロンプトの「絶対ルール」を切り替える  
  例: `csp-strict` なら「scriptタグを出さない」「インラインハンドラ禁止」  
- 使えるブロック一覧もフィルタ後のものだけを列挙

### 3.6 書き出し形式との関係（将来）

プロファイルは「文書の能力」で、HTML/ZIP/PDF は「袋」。  
ただし例えば:

- `safe-static` → PDF/PNG との相性が良い旨を案内  
- `rich` → 静的ホスト手順で注意を出す  

初期は **案内だけ** でよく、形式そのものをロックしなくてよい。

## 4. マイテンプレート（同列）

| 項目 | 案 |
|---|---|
| 保存内容 | `version`, `profile`, `shell`, `blocks`, theme 等（文書スナップショット） |
| 保存先（v1） | localStorage（件数上限あり）。必要なら JSON ファイル書き出し |
| 一覧 | 新規フローの「公式／マイ」タブ |
| プロファイル | テンプレ作成時の profile を保持。開くとき不一致なら確認 |

フォーマット v2 とプロファイルが固まってからの実装が安全だが、**localStorage に v1 文書を名前付き保存**する最小版は先に出してもよい。

---

## 5. 移行・互換

| 方向 | 挙動 |
|---|---|
| v1 → 編集 | shell=`stack` として内部昇格（ユーザには黙ってよい） |
| v1 HTML を開く | 現行どおり |
| v2 HTML を古いエディタで開く | 未知属性は無視。可能なら main region の blocks だけ読むフォールバックを検討 |
| AI が v1 構造を吐く | 引き続き受理（profile は interactive 扱い等） |

---

## 6. やらないこと（この草案の境界）

- 自由配置・レイヤー・ピクセル指定のデザインツール化  
- プロファイルなしで全部品を常時解放したまま CSP 対応を「書き出しオプションだけ」でごまかすこと  
- v1 文書を開けなくする破壊的アップグレード  
- いきなり rich 用 fetch 部品から着手すること（先に shell + profile 骨格）

## 7. 実装フェーズ提案

| Phase | 内容 | 目安 |
|---|---|---|
| **0** | 本メモ合意。`REG` に `needs` を付ける調査（現行部品の仕分け表） | 仕様 |
| **1** | プロファイル選択 UI + パレット／テンプレ／プロンプトのフィルタ（中身はまだ v1 shell） | 実装 |
| **2** | `version:2` + shell=`stack` 明示。マイテンプレ最小 | 実装 |
| **3** | shell=`sidebar`/`topnav` + carousel/tabs のどれか一本 | 実装 |
| **4** | 残り部品・AI完全追従・マイテンプレの入出力 | 実装 |

手触り D1–D4 は Phase 0〜1 と **バッティングしない**（文言・確認ダイアログ層）。
