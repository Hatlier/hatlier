## 0. 名前と意味（確定）

**製品名 / Organization:** Hatlier

| 要素 | 意味 |
|---|---|
| **H** | HTML |
| **at** | 「〜で」「〜に」という自然な接続のニュアンス（Atelier の面影） |
| **lier** | フランス語 *lier* — つなぐ・結ぶ |

**一文の気持ち:**  
つなぐことで創作が生まれる場所。HTMLは完成品ではなく、創作物を包む器。

（旧称 Atelier。`atelier.html` はリダイレクト。書き出し id は `hatlier-doc`。旧 `atelier-doc` も開ける。）

---

## 1. 一言で言うと

**ちょっとしたページを、生成AIなしでも自分で直せる。**

HTMLを「編集できない成果物」ではなく「人が手で触れるフォーマット」として扱い、そのフォーマット管理ができていればデザインツールになる。  
Hatlier はそれを、ブラウザだけで組み立て・書き換え・配布できる **1ファイル完結** の道具にする。

---

## 2. なぜ作るか（問題意識）


| いま起きていること              | それが生む摩擦                                              |
| ---------------------- | ---------------------------------------------------- |
| 生成AIが HTML/CSS を大量に出せる | 出力は「読む・見る」向きで、**非エンジニアが手で直す導線が弱い**                   |
| ノーコード／デザインツールが強い       | 成果物が独自形式に閉じ、**自分の手元・配布・再編集の主導権が薄い**                  |
| Markdown は誰でも書ける       | 一方 HTML は「タグを覚えるもの」と見なされ、**フォーマットとしての手触りが過小評価されている** |
| 作ったページの用途は多様           | 「Webサイト」だけではないのに、道具の目線がサイトビルダーに寄りがち                  |


**打破したい状態:**  
「AIやツールが作ったあと、人が手動で直せず、またツールに戻るしかない」こと。

**目指す状態:**  
ちょっとしたページなら、タグを覚えなくても自分で直して書き出せる。凝ったページは本職と生成AIに任せればよい。

## 3. プロダクトの定義



### 3.1 ツールのポイント（これが本線）

- **ちょっとしたページ**向け（ペライチ・告知・簡単なレポート・自分流の宣伝ページなど）
- **生成AIを使わずに自分で直せる**ことが価値の中心
- **実ユーザの使いやすさ**が最優先（機能数より動線と手触り）
- 成果物は **エクスポートして各自の目的で使う**（サイト公開専用ではない）



### 3.2 Hatlier であるもの

- **フォーマット管理ツール**（見た目の自由度より、壊さない構造と手触りを優先）
- **1ファイル HTML** で完結するプロダクト（アプリ本体も、書き出しの基本形も）
- **手動編集が第一級**（AIは入口の補助線）



### 3.3 やらないこと（明示的に捨てる）


| やらない                       | 理由                         |
| -------------------------- | -------------------------- |
| 凝ったページ・高度なデザイン表現を本職並みにやること | 本職と生成AIに任せる。Hatlierの土俵ではない |
| Figma代替の自由配置キャンバス          | 制約ある部品で「崩れない・直せる」を守る       |
| アカウント必須のホスティング付きサイトビルダー    | 1ファイル・手元完結と衝突しやすい          |
| タグ学習前提のHTML IDE            | 非エンジニアの手動編集が本線             |
| 機能の盛り込み競争                  | 実ユーザの使いやすさを落とすなら後回し／やらない   |
| バックエンド前提のMCPサーバ化を急ぐこと      | いまの原則と合わず、優先度が低い           |




### 3.4 想定ユースケース（ペルソナより先にこれで足りる）

- ペライチ（案内・募集・告知）
- 自分なりのフォーマットの宣伝・紹介ページ
- 簡単なレポート・議事の見える化・社内共有
- 自己紹介ページ
- SNS用の投稿用画像
- AIが吐いたHTMLを開いて、人が仕上げる作業台

**共通点:** 「ちょっとした1枚を、手元で直し、誰かに渡せる」こと。

### 3.5 ペルソナについて（いまは具体化しなくてよい）

詳細ペルソナ（年齢・職種・一日の流れなど）は **後回しでよい**。

理由:

- 本線の価値はすでに「ちょっとしたページを自分で直す」で十分に絞れている
- 早すぎるペルソナは機能リストを膨らませやすい
- 実ユーザの使いやすさは、**渡したあとの摩擦**（どこに置くか・どの袋で渡すか）で見えた方が早い

代わりに、いま使うのは **ジョブの粗い二層**（次節）だけ。  
ペルソナの肉付けは、MVP〜Step 2 で実際の導線を触ったあとでも遅くない。

### 3.6 書き出しの二層（誰が何をうれしいか）

出力の好みは、ペルソナ名より先に次の二層で分かれる。


| 層 | うれしい袋 | 典型の都合 | Hatlier側の答え方 |
|---|---|---|---|
| **1HTML層** | HTML 1枚 | メール添付、チャット送信、社内共有、「ファイル1つ」が楽。画像はdata URL等で内蔵されていてよい | **本線の書き出し**。まずここを最速・最安定にする |
| **ZIP層** | HTML + static（画像・CSS・JSを分けたZIP） | 静的ホストに上げたい、容量・キャッシュ・差し替えを分けたい、ホスト側の制約 | **第二の書き出し**。1HTMLを捨てない。必要になったら並べる |

両方いる、でよい。どちらか一方に寄せて他方を消さない。  
MVPは **1HTMLを完璧に近く**、ZIPは Step 2 で「載せる」とセットで厚くする、が自然。

### 3.7 渡したあとの導線（エクスポート先＝ユーザビリティ）

「書き出せた」で終わらず、**置ける場所への短い導線**があると使いやすい。

方針:

- Hatlier自身がホスト事業者にならない（アカウント必須ホスティングを本線にしない）
- 代わりに、よく使う置き場への **導線・手順・（可能なら）薄い連携** を足す
- 候補の置き場イメージ: Google Drive、登録不要寄りの静的ホスト、登録必要ホストは次点
- 連携の深さは段階的でよい（最初は「こう置く」ガイド → 慣れたらワンショット連携）

これはペルソナより **ジョブ（渡す／載せる）** に紐づく機能なので、Step 2 の主戦場になる。

## 4. 中核思想（設計原則）



### P1. HTML はフォーマットである

成果物の正本は HTML。独自バイナリやクラウド専用データに閉じない。

### P2. 手動編集が第一級

クリックで直す・差し替える・並べる。「再生成しないと直せない」を避ける。  
**生成AIなしでも直せる**のがポイント。

### P3. 制約がデザインを守る

自由配置は薄くする。決まった部品に流し込み、誰が触っても大きく崩れない。

### P4. 1ファイル完結を本線にする

サーバー・アカウント・ビルドなし。本体HTMLを開けば編集できる。  
（外部連携や fetch 部品は「後段の拡張」として本線を壊さない形で足す。）

### P5. AI は入口、人が出口

プロンプトで Hatlier 形式を作らせる → 貼る／開く → 人が仕上げる。  
凝ったものはAIに任せ、Hatlierは「直す・渡す」に集中する。

### P6. 用途はユーザーが決める／使いやすさが最優先

公開先・埋め込み先・出力形式はユーザー側の都合に合わせる。  
機能追加の判断基準は「実ユーザが迷わず直せて書き出せるか」。

### P7. やらないことを守る

アイデアは駐車場に置き、MVPとステップを歪めない。

## 5. 現状（実装のスナップショット）

リポジトリの中心は `hatlier.html`（1ファイル完結）。旧 `atelier.html` はリダイレクト用。


| 領域     | 内容                                                                                               |
| ------ | ------------------------------------------------------------------------------------------------ |
| ブロック編集 | ヒーロー、テキスト、カラム、統計、ステップ、表、グラフ、ToDo、トグル、コールアウト、FAQ、タイムライン、バッジ、ボタン、区切り、引用、コード、リンクカード、画像、動画、PDF、締め など |
| デザイン   | テーマ配色・背景パターン・フォント                                                                                |
| 操作     | インライン編集、並べ替え、部品パレット、Undo/Redo、テンプレートから新規                                                         |
| 入出力    | HTML書き出し／開く、任意HTMLの取り込み（フリー編集）                                                                   |
| 保存     | localStorage への自動保存（明示的な「途中保存」UIは薄い）                                                             |
| AI連携   | Hatlier形式を出力させるプロンプトの提示・コピー                                                                      |
| ランタイム  | ブラウザのみ、外部ライブラリ依存なし                                                                               |


**二つの編集モード**

1. **ブロック編集** — 構造付き。再編集性が高い。本線。
2. **フリー編集** — 任意HTMLを文字・画像・リンクだけ安全に編集。

## 6. フォーマット契約（議論の核）



### 6.1 ブロックモード

- ドキュメント属性（テーマ等）+ ブロック列
- 決まった `class` / `data-*` / タグ構造
- 編集箇所は `data-edit` などで明示
- 書き出しと「開く」で往復できる



### 6.2 フリーモード

- 取り込んだ HTML が正本
- 触れる範囲はテキスト・画像・リンクに限定



### 6.3 未決

- [ ] フォーマットのバージョン明示（`data-hatlier-version` 等。移行期は旧名も可）

- [ ] コミュニティ部品の登録規約（後段の fetch 部品含む）
- [ ] フリー → ブロックへの昇格を目指すか（難易度高・MVP外）

### Step 1 — MVP（直近でやる）

**定義:** 1HTMLのプロダクトのまま、**実ユーザが迷わず直して、1HTML層として渡せる**ところまで整える。


| 項目 | 内容 | 状態 |
|---|---|---|
| A. AI→貼り付け動線 | 「貼る」で厳密判定（ブロック or フリー） | **実装済** |
| B. 途中保存の明示 | 「保存」＝ localStorage（⌘S） | **実装済** |
| C. 書き出しメニュー | HTML / ZIP / PDF / PNG / WebM / スライドWebM | **実装済（v0.3.0）** |
| D. 手触り・動線の研磨 | ラベル・失敗時・初回案内 | 残 |
| （横断）バージョン＋予定の見える化 | 画面 `vX.Y.Z` → [`ROADMAP.md`](./ROADMAP.md) | **実装済** |

#### 出力ラダー

| 段 | 形式 | 状態 |
|---|---|---|
| 1 | HTML 1枚 | 済 |
| 2 | ZIP（HTML+static） | 済 |
| 3 | PDF（印刷→PDFに保存） / PNG / WebM短尺 | 済（v0.3.0） |
| 4 | スライド動画（ブロック・パラパラ WebM） | 済（v0.3.0）。AGIF/MP4優先は Next |

#### A の設計方針（多様性 × 厳密フォーマット）

AI出力は多様で、Hatlierのブロック契約は厳密。**曖昧な自動変換で半端なブロック文書を作らない。**

| 判定 | 結果 |
|---|---|
| `hatlier-doc` / `atelier-doc` あり | ブロック編集 |
| `.blk` 構造が parse できる | ブロック編集 |
| それ以外 | フリー編集（文字・画像・リンク） |
| （前処理のみ） | ` ```html ` フェンスを外すだけ |

#### B の設計方針（保存先）

| 層 | 手段 | いつ |
|---|---|---|
| **いま（MVP）** | localStorage（自動＋明示「保存」／⌘S） | 同一ブラウザで続きから開く安心感 |
| **配る・残す本線** | HTML書き出し（1ファイル） | 別端末・他人・長期保管 |
| **あと（Step 2）** | Drive / 静的ホストへの導線・薄い連携 | 「置く」摩擦を減らすとき |

**Driveは技術的には可能**（ブラウザから Google Identity + Drive API。ただし OAuth クライアントID・同意画面・Org側のAPI設定が要る）。  
1ファイル完結・ゼロ設定の思想とは摩擦が大きいので **MVPではやらない**。「保存」＝ブラウザ内、「渡す」＝HTML書き出し、と役割を分ける。


**MVPでやらない（明示）**

- ミニゲーム／占い等のエンタメ部品（→ Step 3+、fetch追加）
- 解析ツール一式（GA4/Umami/Plausible）の本実装（→ Step 2）
- ホスト連携・Drive連携の本実装（→ Step 2。「置く」導線とセット）
- **ZIP（HTML+static）の本実装**（→ Step 2。ZIP層向け。1HTMLと併存）
- 動画・パラパラ漫画書き出し（→ 需要が見えてから）
- 詳細ペルソナ設計（→ 実導線を見てからでよい）
- ローカルAI・重いメディア編集・CSP厳密プロファイル（→ 駐車場／後段）
- MCPサーバ化（→ 原則外・低優先）

**MVPの完了イメージ**

> 非エンジニアが、AI出力を貼る or テンプレから始めて、文字を直し、保存した気持ちになって、**HTML 1枚で渡せる**。

### Step 2 — 渡す・載せる（二層の書き出し＋置き場の導線）

MVPで「直せる／1HTMLで渡せる」ができたあと、**ZIP層**と**置き場**の摩擦を減らす。


| テーマ | 候補 | メモ |
|---|---|---|
| ZIP層の書き出し | HTML+static の ZIP | **メニュー実装済（v0.2.0）**。スマホ向け導線は続く |
| 置き場への導線 | Google Drive、静的ホスト（登録不要寄りを先、登録必要は次点） | 自前ホストにしない。最初は手順ガイドでも価値がある |
| 薄い連携 | Drive保存、Tiiny Host 等のワンショット | 導線で足りなければ連携を足す |
| 解析の差し込み | GA4 / Umami / Plausible | 「載せたあと測りたい」層向け。無理な先は明示 |
| マイテンプレート | 自分の構成を雛形として残す | 直しやすさの延長 |
| 機能別バージョン | 用途で薄いプロファイル分け | 必要になってから |


**ここでもやらない:** 凝ったページ向け表現の深掘り、ゲーム部品の本体肥大化、アカウント必須を本線化すること。

---



### Step 3 — 部品の拡張（コアは同梱、遊びは fetch）


| 層       | 方針                       | 例                                  |
| ------- | ------------------------ | ---------------------------------- |
| コア部品    | 1ファイルに残してよい、編集・伝達に直結するもの | タブ、カルーセル、読了目安、QR、ページ内検索、時計 など（要取捨） |
| 拡張部品    | **fetch で追加**（本体を太らせない）  | ミニゲーム、占い各種、今日は何の日、RSSニュース など       |
| 書き出しの遊び | スライドWebMは v0.3.0 で実装。AGIF / MP4優先は ROADMAP Next |


拡張部品の原則:

- オフライン必須の本線を壊さない（未取得時は壊れたように見せない）
- 著作権・出典が絡むもの（ニュース等）は仕様で慎重に扱う
- 「面白いから入れる」より「実ユーザのページに載る理由があるか」

## 8. OSS・リポジトリ戦略

### 8.1 公開の狙い

- Organization **Hatlier** 配下で公開。使ってくれる人がいればうれしい
- 部品・テンプレ・テーマは提案・PRで増やせる形が望ましい
- **最初のCTA案:** 「ちょっとしたページを直して書き出してみて」→ 感想。部品提案はその次
- 守りたいこと: 1ファイル本線、軽量、制約の質、セキュリティ、**やらないこと**

### 8.2 リポジトリ戦略（いまの推奨）

**結論:** いまは **`Hatlier/hatlier` に、配布物としての `hatlier.html`（1ファイル）を置く**。  
`editor/` + `src/` + `package.json` 構成は、**いまの段階では採らない**。

理由は「将来どうしたいか」より、**いま守る原則**に直結するから:

| 原則 | `hatlier` + 1HTML | `editor/` + src 分割 + ビルド |
|---|---|---|
| サーバー・ビルドなしで動く | そのまま満たす | 開発者向け手順が増える（利用者向けにまた1ファイルへ束ねる必要） |
| 実ユーザが clone / 開くだけ | READMEが短い | `npm install` が前に立つ |
| OSS初期の見通し | 入口が1つ | 空の `components/` が先に目立つ |

`src/` 分割がよくなるのは、例えば次の**合図**が来たとき:

- 本体が巨大化し、部品ごとのレビュー・テストが必須になった
- 複数人が並行でブロックを量産し、単一ファイル衝突が常態化した
- ビルドして1HTMLを吐くパイプラインを、開発者体験のために導入する決断をした

そのときはリポ名を `hatlier` のまま中を `src/` 化してもよいし、実装専用に `editor` を切ってもよい。**リポ名の悩みと、ソース分割の悩みは別問題**として扱う。

| リポ名の候補 | いま | メモ |
|---|---|---|
| **`Hatlier/hatlier`（採用）** | ✅ | アプリ名＝Org＝リポ。探しやすい |
| `Hatlier/editor` | 見送り | 役割名。製品名が薄くなる。分割が現実になってからでも可 |

**最初に作らないもの:** 空の `blocks` / `templates` / `themes` / `website` / `specs`

**後から割る合図**

| 合図 | 切り出し先の例 |
|---|---|
| fetch部品が独立版になり周期がずれる | `Hatlier/blocks` など |
| 公式サイトが別周期 | `Hatlier/site` |
| フォーマット仕様だけが外部参照される | `Hatlier/format` |

### 8.3 Org の見え方（最小セット）

いま用意するもの:

1. **`hatlier`** — 本体（`hatlier.html`）
2. Org の Description / README（Orgプロフィール）に一文ピッチ
3. （任意）`Hatlier/.github` — デフォルトの Issue／PR テンプレ

## 9. 成功の見方（仮説）

- 生成AIなしで「直して書き出せた」
- ちょっとしたページ用途で、凝ったデザインツールに逃げなくてよい
- 書き出しがメール添付・静的ホスト・社内共有でそのまま使える
- （Step 2以降）他サービスや解析・ホストに載せやすい、と言われる
- 「多機能」より「直しやすい」と語られる

## 10. 今すぐ決めたい議論アジェンダ

1. **MVPは1HTMLに寄せる合意でよいか**  
   PDF/PNGは任意。ZIPとホスト導線は Step 2（二層のもう一方＋置き場）。
2. **AI貼り付け動線の形**  
   「開く」の拡張か、専用の貼り付けモーダルか、クリップボード検知か。
3. **途中保存のUX**  
   自動保存との関係、保存済み表示、容量オーバー時の案内。
4. **Step 2の最初の一本**  
   ZIP書き出しか、Drive／静的ホストへの導線（ガイドだけでも可）か、解析スニペットか。
5. **コア同梱 vs fetch**  
   どの部品までを本体に残すか（タブ等は同梱候補、占いは fetch）。
6. **一文ピッチ**  
   例: 「HTMLをつないで、ちょっとした創作物を自分の手で直して渡せる場所」／「ちょっとしたHTMLページを、自分の手で直して渡せる道具」
7. **ペルソナは後でよいか**  
   いまはジョブ二層（1HTML／ZIP）とユースケース列挙で進める。
8. **メインリポ名**  
   **`Hatlier/hatlier` で進める**（`editor` + `src/` 分割は、ビルド導入の合図が来るまで見送り）。

## 11. 用語


| 用語 | 意味 |
|---|---|
| ちょっとしたページ | 凝ったサイトではない、1枚〜数ブロックで用が足りる成果物 |
| ジョブ二層 | 詳細ペルソナの代わりに使う、1HTML層／ZIP層という渡し方の違い |
| 1HTML層 | ファイル1枚で渡せるとうれしいジョブ |
| ZIP層 | HTML+staticでホスト等に載せるとうれしいジョブ |
| 置き場導線 | Driveや静的ホストなど、書き出し後に「置ける」ようにする案内・連携 |
| フォーマット | 人が触れて往復できる構造付き表現（主にHTML） |
| ブロック | 編集と見た目の単位になる決まった部品 |
| 書き出し | 配布・再利用可能なファイルを出力すること |
| fetch部品 | 本体に同梱せず、必要時に取得して足す拡張部品 |
| 駐車場 | 思いつきを捨てず、ロードマップを歪めない退避場所 |

## 付録 C. アイデア駐車場（優先度なし・思いつき保管）

> 実装コミットではない。Step判定を通したものだけ上のロードマップへ昇格する。



### C1. 動線・保存・テンプレ

- AIプロンプトで作ったHTMLをそのまま貼って Hatlier 形式表示 ← **MVP候補**
- 途中保存ボタン（localStorage等）← **MVP候補**
- マイテンプレート
- 機能別バージョン分け公開



### C2. エクスポート

- 1HTML（HTML）← **本線・1HTML層・MVP優先**
- ZIP（HTML+static）← **ZIP層・Step 2**
- PDF / PNG一枚絵 ← もう1形式候補（全部は入れない）
- スライド区切りパラパラ（AGIF / MP4 / webm）
- スマホ向け: ZIP DL、フォルダ内包DL



### C3. 外への載せ方・連携（置き場導線）

- Google Drive（保存・共有の導線／薄い連携）← Step 2候補
- 静的ホストへの導線（登録不要寄りを先、Netlify等は次点）← Step 2候補
- GA4（ヘッダ差し込み。Notionは制約あり）
- Umami / Plausible
- Notion（ホスト用）/ Figma（ステップアップ）/ Tiiny Host / Netlify
- 埋込URL（X-Frame-Options, CSP, Cookie, JWT, 権限, origin制限）
- MCPサーバ化（バックエンド必要 → 低優先・原則外）



### C4. 欲しい部品（コア候補になりうるもの）

- タブ分け / カルーセル / トグル（既存あり）/ パラパラ漫画
- ワードクラウド / アニメーション表現
- アナログ・デジタル時計
- 音声録音 / QR・バーコード
- 動画の長さ編集 / 画像のちょい編集
- サイドバー / ページ内検索 / 読了時間目安
- codeオプション（iframeで別HTML・アンケート埋込、Python実行、JS実行）
- JS禁止モード（CSP厳格）
- ローカルAI（tflite/onnx、MobileNet、YOLO、音声認識、supertonic 等）
